Kubernetes服务概述

Kubernetes Pods是会终止的。Pods实际上有一个 生命周期。当一个工作节点结束时，运行在Pods上的Node也会丢失。 一个 ReplicaSet 会通过创建新的Pod来动态地驱动集群回到原始的状态，以保持应用程序运行。另一个例子，考虑到一个运行中的镜像背后有三个维持。这些副本是可替换的；前端系统无需关心后端的副本，即使Pod丢失和重建。也就是说，Kubernetes集群中的每个Pod都有唯一的IP地址，即使是同一Node的Pods。因此需要一种方法来动态地协同Pod之间的变化，以便应用程序继续运行。

Kubernetes中的Services是一种抽象化，它定义了Pod的逻辑集和访问Pod的策略。Services使互相依赖的Pods实现松耦合。如同所有的Kubernetes对象一样，Service由YAML Using Source IP 教程。另外也可以参考 Connecting Applications with Services.

另外，请注意某些Services的用例通常没有涉及定义selector。没有selector创建的服务也不会创建相应的Endpoints对象。这使用户可以将Service手动映射到指定端点。另一种没有使用selector的情况是，你仅使用了类型: ExternalName。

总结

  • 将Pods暴露给外部通信
  • 跨多个Pods负载平衡流量
  • 使用标签

Kubernetes服务是一个抽象层，它定义Pods的逻辑集合并使Pods能被外部暴露通信、负载均衡和服务发现。


Services and Labels

服务通过一组Pods来路由通信。Services是一种抽象化，它能够在Kubernetes内终止和复制pods而不影响你的应用程序。互相依赖的Pods（比如应用程序中的前端和后端组件）间的发现和路由被Kubernetes Services处理。

Services通过 标签和选择器匹配一组Pods，分组意味着允许对Kubernetes中的对象进行逻辑操作。 Labels作为键值对被附加到对象，并以多种方式来使用：

  • 指派对象用于开发，测试和生产
  • 嵌入版本标签
  • 使用标签对对象进行分类

你可以在创建一个Deployment时创建一个Service通过使用
--expose in kubectl.



标签可以在创建时或以后附加到对象。标签支持随时修改。让我们现在通过Service来暴露我们的应用程序并且应用使用一些标签。
